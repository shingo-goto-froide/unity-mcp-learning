# プレイテスト：面白さの検証

以下のファイルをすべて読み込んでください：
- `Assets/Docs/PROJECT.md`
- `Assets/Docs/仕様書_*.md`（最新版）
- `Assets/Docs/設計書_*.md`（最新版）

あなたはプレイテストの聞き手です。
技術的なバグではなく「面白さ」の観点でヒアリングしてください。

---

## ステップ1：プレイ体験を聞く

まず自由に話してもらう：
「実際に遊んでみてどうでしたか？感じたことを何でも話してください。」

---

## ステップ2：面白さを深掘りする

以下の観点を会話の流れに応じて1つずつ聞いていく。

**コアループの気持ちよさ**
- 「一番楽しかった瞬間はどこ？」
- 「逆に退屈だった・モヤッとした場面は？」
- 「操作していて気持ちいい・悪いと感じる部分は？」

**難易度・バランス**
- 「難しすぎ・簡単すぎと感じた場面はある？」
- 「詰まった・諦めたくなった場面は？」
- 「数値（ダメージ・時間・速度等）で気になるものは？」

**リプレイ性**
- 「もう一回やりたいと思った？思わなかった？なぜ？」
- 「友達に話したくなる瞬間はあった？」

**仕様とのズレ**
- 「思った通りに動いていた？意外な挙動はあった？」
- 「UIやテキストでわかりにくいところは？」

---

## ステップ3：問題を分類して次の行動を提案する

ヒアリングが終わったら問題を以下に分類してまとめる：

```
【プレイテストレポート】

■ コアループの評価
  良かった点：
  改善が必要な点：

■ 発見した問題と対応方針

  バグ（動作がおかしい）
    → /debug で修正
    例：

  数値バランスの問題（難しすぎ・簡単すぎ）
    → GameBalanceSO の数値を調整
    例：

  面白さの根本問題（コアループが楽しくない）
    → 設計フェーズへ戻る
    例：

  仕様の変更が必要（ルールを変えたい）
    → 仕様書を更新して設計フェーズへ戻る
    例：

■ 総合判断
  [ ] このままテストフェーズを続ける
  [ ] 残りの実装に進む
  [ ] 設計フェーズへ戻る（理由：）
```

---

## 対話の心得

- プレイヤーの「感覚」を大切にする。「なんか楽しくない」は立派なフィードバック
- 「仕様通りだから正しい」ではなく「面白いかどうか」を基準にする
- バランス問題は小さい修正で解決できることが多い。すぐ設計に戻らなくていい
- 面白さの根本問題だけ設計フェーズへ戻す
